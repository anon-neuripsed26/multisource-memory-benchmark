# Anonymization Notes (Double-Blind Submission)

This document records the anonymization choices made for the NeurIPS 2026
Evaluations & Datasets Track double-blind submission, and the steps that
will be reversed at camera-ready.

## Status

- Audit script: [`scripts/anonymization_audit.py`](scripts/anonymization_audit.py)
- Last clean run: **0 email / GitHub-owner / ORCID findings across
  58,000 files** (run via `python3 scripts/anonymization_audit.py
  --root .` against the release tree, including
  `tests/`, `examples/`, `_archive/`, and pytest caches).
  Note: `ANONYMIZATION_FORBIDDEN` in the audit script is a
  template tuple intentionally empty in this template; authors
  populate it locally with their own name/handle/affiliation strings
  before running pre-camera-ready review. The 0-finding number reflects
  the structured-identifier categories only.
- All identifying material in this directory has been replaced with the
  literal placeholder `Anonymous`.

## Anonymization risks the audit does NOT catch

The audit covers structured identifiers (emails, GitHub owners, ORCID).
It does NOT catch:

- Distinctive project codename collisions with prior public artifacts.
- Internal-jargon checkpoint identifiers that may pattern-match a known
  group's release-tracking conventions.
- Author-style giveaways in long-form prose.
- Git history (when shipping as a clone, not a tarball) — strip
  `.git/` before publishing to the anonymous GitHub code mirror.
- Absolute filesystem paths in committed logs (`/Users/<name>/...`).
- Timezone / locale strings that narrow geography.

Proofread Tier B documents and CITATION.cff / pyproject.toml manually
in addition to the automated audit before submission.

| Surface | Submitted (anonymous) | Camera-ready (final) |
|---|---|---|
| `CITATION.cff` `authors` | `Anonymous Anonymous`, `Anonymized for double-blind review` | Real author + ORCID + affiliation |
| `CITATION.cff` `repository-code` | `https://github.com/anon-neuripsed26/multisource-memory-benchmark` | Real GitHub URL |
| `pyproject.toml` `[project].authors` | `name = "Anonymous (NeurIPS 2026 submission)"` | Real author + email |
| `README.md` Anonymization section | "anonymized for double-blind review … mirror at `github.com/anon-neuripsed26/multisource-memory-benchmark`" | Removed; replaced with real repository, citation, and contact |
| `DATASHEET.md` Maintainer | "anonymized for review" | Real maintainer name + contact |
| Hugging Face dataset URL | `anon-neuripsed26/multisource-memory-benchmark` | Final org / dataset name |
| Zenodo record | not minted (no DOI yet) | Final DOI |

## What is NOT anonymized (and why)

- **Apache-2.0 / CC-BY-4.0 license boilerplate**: standard text, contains
  no identifying information. The `[yyyy] [author]` placeholder in
  `LICENSE` is the verbatim Apache template and is not personally
  identifying.
- **Third-party platform handles** (Hugging Face, MLCommons, NeurIPS,
  OpenML, Kaggle, etc.): these are organization / project handles, not
  the authors. They are explicitly allow-listed in the audit script.
- **Generic / example email addresses** (`info@example.com`,
  `noreply@github.com`, `support@*`): allow-listed.

## Audit policy

Run the audit at any time:

```bash
python3 scripts/anonymization_audit.py --root .          # report only
python3 scripts/anonymization_audit.py --root . --strict # exit 1 on any finding
```

The audit fails (with `--strict`) on any:

1. Email address not in `EMAIL_ALLOWLIST`.
2. `github.com/<owner>/...` URL whose `<owner>` is not in
   `GITHUB_OWNER_ALLOWLIST`.
3. ORCID iD pattern.
4. Any string in `ANONYMIZATION_FORBIDDEN` (kept empty in this public
   template; populate locally with concrete real names you want
   double-checked, then revert before pushing).

CI runs the audit in non-strict mode (warnings printed) on every PR; the
release tarball build step runs it in `--strict` mode and refuses to
package if any finding is present.

## Reviewer access (anonymous code mirror)

For NeurIPS 2026, the double-blind code mirror is hosted at:

```
https://github.com/anon-neuripsed26/multisource-memory-benchmark
```

(Also linked from the OpenReview submission.)

The Hugging Face benchmark record will be published under an
`anonymous-*` Hugging Face organization without author affiliation, then
moved to the final organization name at camera-ready. The Croissant
metadata is auto-generated by Hugging Face from the dataset upload
(accessible at the dataset's `/croissant` endpoint); de-anonymisation
flows automatically through the HF dataset metadata once the real
author attribution is set on the HF organization page.

## Camera-ready reversal checklist

When de-anonymizing:

- [ ] Restore real author block in `CITATION.cff` (both top-level and
      `preferred-citation`).
- [ ] Update `repository-code` in `CITATION.cff` to the final GitHub URL.
- [ ] Update `[project].authors` in `pyproject.toml`.
- [ ] Update `README.md`: replace the "Anonymization" section with a
      "Citation" section and project contact information.
- [ ] Update `DATASHEET.md` maintainer field.
- [ ] Update `data/fetch_benchmark.py` `HF_REPO_ID` constant to the
      final non-anonymous Hugging Face org/dataset name.
- [ ] Mint a Zenodo DOI for the dataset, add the DOI badge to
      `README.md`, and reference it from the HF dataset card so it
      flows into the auto-generated Croissant.
- [ ] Re-run `python3 scripts/anonymization_audit.py --root .` — at this
      point the audit is *expected* to find your real email / GitHub
      handle. Either remove `--strict` enforcement from CI, or move the
      now-public identifiers into the allow-list.
- [ ] Re-fetch and verify the HF `/croissant` endpoint for the
      de-anonymised dataset via the
      [Croissant checker](https://huggingface.co/spaces/JoaquinVanschoren/croissant-checker).
      Only generate a local Croissant file if HF auto-generation is
      insufficient (in that case, ship a real artefact, not a placeholder).
